# Radix — State of the project (2026-06-09)

This log builds on `notes-20260608.md`, which has the full background. The short version
of *why* this project exists: Radix is a prototyping tool where a user describes an app,
refines it with an agent, and gets a working prototype in the browser. Everything the real
app would talk to (databases, payments, other users, hardware) is faked convincingly. The
core insight is that **a prototype is just JavaScript surrounded by a library of fakes** —
build the library once, and the agent only writes the genuinely app-specific parts. Two
co-equal foundations: the **data foundation** (schema → fake db → auto-CRUD) and the
**world foundation** (a simulator that generates events over time, so reactive apps have
something to react to). The second one is what separates Radix from yet another
CRUD-to-UI generator.

## What changed since yesterday

Yesterday's log was written mid-stream; the commit that landed last night (`b4e6b1d`,
"Add console and db schema") actually went further than the log describes. Two of the
"best next steps" from that log were started the same evening:

### 1. `db.define()` — first step toward the schema-driven store (Phase 2)

The db now accepts a declarative schema:

- `db.define(schemaDef, opts)` takes `{ collection: { fields, seed } }`. Fields carry a
  type (`string`, `number`, `boolean`, `enum` with `values`), optional `default`, and
  optional `required`. A shorthand string form (`field: 'number'`) normalizes to the full
  shape.
- `create` and `update` now validate against the schema. In strict mode violations throw;
  otherwise they log a warning. Defaults are applied on create.
- Seeding moved into the schema: each collection declares its `seed` rows, and `define`
  installs the replayable seed function. This answers the "seeding's home" open question
  from yesterday — it's schema-driven now, not an imperative app call. `db.__seed` still
  works but is explicitly legacy.
- `db.schema()` exposes the registered schema, and the shell can request it over the
  bridge (`radix:db:schema`).
- **All twenty examples were migrated** from `__seed` to `db.define` (that's the wide diff
  across `src/tests/examples/` in the commit).

This is deliberately not the full Phase 2 engine — there's no relations vocabulary in the
schema yet, no indexes, no auto-CRUD API derived from it. But the strict-core shape from
`plan.md` (typed fields, enums, defaults, required) now exists and is exercised by every
example.

### 2. The simulation console grew real controls (Phase 4)

The shell's inspector panel is now a proper (if small) console:

- **Clock controls** — play / pause / step from the shell, via new bridge messages
  (`radix:clock:play|pause|step|get`). The runtime pushes clock state to the shell on
  every change, so the controls stay in sync without polling.
- **Log panel** — the runtime streams `log` entries to the shell (`radix:log:entries`),
  stamped with simulated time. Headless examples (cron, slackbot) are now drivable and
  observable entirely from the shell.
- **Schema view** — the inspector can show a collection's declared fields, not just its
  rows.

This matters beyond convenience: it validates the postMessage bridge design (command in,
data out, plus push-updates for state that changes continuously) before the real Phase 1
runtime is built on it.

### 3. Batch 3 example ideas (uncommitted)

`src/tests/example-apps.md` has an uncommitted addition: five "boundary case" apps
(41–45) chosen because each one stresses a part of the design nothing else does:

- **Spreadsheet with formula engine** — pure-compute + dependency DAG + simulated
  collaborators.
- **Event-sourced ledger** — append-only data; tests whether the db can support an
  immutable pattern at all.
- **Tower-defence game** — frame-loop spine at scale; hundreds of per-tick actors.
- **Property-graph explorer** — multi-hop/recursive queries that `query(collection,
  filter)` can't express.
- **LLM pipeline builder** — the most direct stress on the stub/graceful-degradation
  design.

These aren't built; they're the targets list for finding where the current contract
breaks.

## Where that leaves the open questions

From yesterday's list:

- ~~Seeding's home~~ — **resolved** by `db.define`. Seed data is schema-declared.
- **State machines / field-write-ownership** — still convention-only. But the schema now
  has an `enum` type, which is the natural place to hang legal-transition metadata when
  we decide to. The hook exists; the enforcement doesn't.
- **The resume rule** — still untested. No example has been reloaded mid-workflow to
  verify the simulator resumes from persisted state instead of replaying from seed-zero.
- **`Math.random()` blocking** — still not done.
- **Relations** — the new schema vocabulary has fields but *no relations section*. This
  is now the most visible gap in the strict core, and the property-graph example (44) was
  added partly to force it.

## Best next steps

1. **Commit the Batch 3 additions** — the example-apps.md change is sitting in the
   working tree.
2. **Test the resume rule now.** The console makes this easy for the first time: load
   food delivery, fast-forward mid-delivery, reload the iframe, and watch whether the
   driver teleports. This is a five-minute test that either confirms or breaks a core
   persistence design decision, and it gates the Phase 1 contract freeze.
3. **Add relations to the schema vocabulary.** `db.define` has fields and enums but no
   `belongsTo` / `hasMany`. The plan's strict-core sketch includes them, `include` already
   exists in the query layer, and no example exercises real relational lookups. One
   genuinely relational example (or the property-graph explorer) plus a `relations` block
   in the schema would close the loop.
4. **Add the event injector to the console.** Clock controls and the log are in; the
   injector ("pretend a payment webhook just arrived") is the next Phase 4 item and needs
   only one more bridge message type.
5. **Then the Phase 1 freeze.** The remaining blockers are the same as yesterday minus
   seeding: the `host` interface shape, `Math.random()` containment, and writing down the
   bridge protocol (which now has enough real message types to document properly).

The agent (Phase 10) stays deferred. The test from yesterday's log still applies: when
building a new example feels like "mostly configuration," the library is ready for an
agent. `db.define` moved the needle — seed + schema is now declarative — but actors and
services are still hand-coded per app.
