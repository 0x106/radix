# Radix — State of the project (2026-06-08)

## What we're building

Radix is a prototyping tool for software. A user describes what they want to build, talks it through with an agent, and gets a working prototype in the browser — fast. Everything the real app would talk to (databases, payment processors, other users, hardware) is faked convincingly enough that the prototype code runs as if it were real. The goal is to help people explore, test, and critique an app _before_ anyone writes production code.

The core insight: **a prototype is just JavaScript surrounded by a library of fakes.** Build the library once, reuse it. The agent's job is to configure the library and write the genuinely app-specific parts.

Two co-equal foundations drive the whole thing:

1. **The data foundation** — a schema-driven fake database and auto-CRUD API. For screen-on-a-database apps.
2. **The world foundation** — a simulator that generates events over time, so reactive apps have something to react to. For chat apps, delivery trackers, job queues, bots — anything that sits idle until something happens to it.

This two-foundation split is the most important architectural decision. Tools that only build the data foundation end up as just another CRUD-to-UI generator. The world foundation is what makes Radix genuinely useful for the half of software that isn't a screen on a database.

## What's been built

### The runtime (Phase 0 — done)

A working prototype of the runtime contract, discovered by hand-building real apps against it. The runtime lives in `src/lib/` and is explicitly labelled "throwaway" — it exists to make the contract visible, not to be the final implementation.

The contract (`src/lib/contract.ts`) defines the shape of `window.radix`, the global handle every prototype uses. These are the handles we discovered actually work:

- **`db`** — the schema-driven entity store. `create`, `update`, `delete`, `get`, `query`, `subscribe`, `reset`. The `subscribe` call fires immediately with current rows, which turned out to be the single most useful affordance — makes React's `useEffect` pattern a one-liner.
- **`events`** — a pub/sub bus. Dead simple: `subscribe(topic, cb)` and `publish(topic, payload)`.
- **`clock`** — simulated time. `play`, `pause`, `step`, `fastForward`, `setTimeout`, `subscribe`. All delayed work routes through this, never the real `setTimeout`, so the whole app can be paused, stepped, or fast-forwarded.
- **`random`** — seeded randomness. `random()`, `int(min, max)`, `pick(arr)`. Same seed always produces the same sequence.
- **`log`** — emits to a simulation console. Entries are stamped with simulated time, not wall time.
- **`actor`** — the world-simulator primitive. A stateful, clock-driven, reactive process. Can tick on a schedule, react to events, or both. This is how the food delivery driver moves, how the chat "other person" responds, how the cron worker fires.
- **`services`** — a stub for external service fakes (email, SMS, payment). Delays are clock-based so they respect pause/step/fastForward.

The runtime implementation (`src/lib/runtimeSource.ts`) ships to the iframe as an inlined browser-JS source string. The app shell injects it before the prototype runs. The db layer persists to IndexedDB, namespaced per prototype, so fake data survives a page reload.

### The app shell (done)

A Next.js app (`app/`) with:

- Auth via InstantDB (magic link).
- A sidebar that lists projects owned by the logged-in user.
- An iframe that renders the selected project's HTML bundle.
- A DB inspector panel (toggle with the "DB" button) that talks to the prototype via `postMessage` — shows all collections and rows, and lets you reset to seed.

Projects are stored in InstantDB. Each project's HTML bundle (a self-contained document: runtime shim + prototype code) is stored in InstantDB Storage and served directly into the iframe. The schema (`app/src/instant.schema.ts`) is straightforward: `$users`, `$files`, and `projects`, with links between them.

### The examples (done)

Twenty working examples (`src/tests/examples/`) covering a wide range of app types. They're deployed to the app via `src/tests/push-examples.ts`, which generates the HTML for each and uploads it to InstantDB Storage under the owner's account.

Current examples:

- **Smoke tests**: counter, todo (plain React, no runtime)
- **Phase 0 / runtime contract**: habits (CRUD), chat (reactive world-sim), cron (headless, clock-driven)
- **DB-focused**: admin dashboard, personal finance, e-commerce shop
- **World engine**: smart home (device fleet), food delivery (multi-actor, services)
- **Batch 2**: task board, trivia, social feed, ride-share, DevOps monitor, onboarding/KYC, booking, Slack bot, music player, API dashboard

The food delivery example is the most complete stress test. It runs two concurrent actors (restaurant + driver), uses cross-collection queries, coordinates actors via db state, and exercises clock-based delays at multiple points in a single workflow.

Three planning documents that are worth reading in order:

- `runtime-contract.md` — findings from Phase 0; the actual contract that emerged from building the examples.

## What the examples taught us

The three Phase 0 examples (habits, chat, cron) and the subsequent batch validated the runtime contract and surfaced several things to carry forward:

**What held up well:**

- `db.subscribe` firing immediately with current rows is the right shape. Every example used it.
- The `clock.setTimeout` routing everything through simulated time worked cleanly — pause/step/fast-forward actually controls the whole app.
- The `actor` primitive (start, tick, on-event) covered every world-simulator case without needing extension.
- The overall architecture — prototype code runs in an iframe against `window.radix` — is clean and worked from the first example.

**What's still open:**

- **State machines and field-write-ownership.** Every reactive app has a status enum that's really a state machine (e.g. order: `placed → accepted → preparing → ready → delivered`). The db can store the enum but can't enforce the transitions. Similarly, some fields have exactly one legitimate writer (only the driver actor updates driver position). Currently these are conventions. Worth first-class support eventually — they recur in every reactive app.
- **The resume rule.** When a prototype persists state and a world-simulator actor also influences that state, reloading must restore persisted state first, then resume the simulator from it — not replay from seed-zero. (Otherwise the driver teleports.) The examples don't exercise this yet because the db uses IndexedDB which does persist, but no example has been tested mid-flight. Re-test this the moment an example is loaded mid-workflow.
- **Seeding's home.** Currently examples call `db.__seed(fn)` which is an example hack. Seed data declaration needs a real, first-class design — probably manifest-driven.
- **`Math.random()` blocking.** The plan says to block real `Math.random()` in the sandbox so prototypes are forced to use `radix.random`. We haven't done this yet because React itself may use `Math.random()` internally. Need a targeted approach (block it in the prototype's own module scope, or lint generated code).
- **Relations haven't been exercised by a true relational example.** The `include` query argument was added, but no example yet relies on real relational lookups in a way that would stress the design.

## What's not built yet

Roughly mapping to the plan phases:

- **Phase 1: Frozen runtime harness.** What exists is explicitly a Phase 0 throwaway. The real runtime needs: a postMessage bridge instead of a shared global, a proper versioning story, the `host` shell boundary (none of the examples exercise this — they all fake the console _inside_ the prototype rather than using `host`), and proper `Math.random()` containment.
- **Phase 2: Generic schema-driven store.** The current db is hand-written, in-memory, without any schema concept. A real implementation reads a schema definition and configures itself — one engine for all apps.
- **Phase 3: World foundation.** The `actor` primitive is a start, but it's single-actor hand-coding. The real engine needs configurable actor patterns, proper composition, and the "action that spawns an actor" pattern formalized.
- **Phase 4: Simulation console.** The DB inspector panel is a narrow start. A real console needs clock controls (play/pause/step/fast-forward exposed in the shell, not just inside the prototype), an event injector, an event log, and failure-injection controls.
- **Phase 5: Service mock shelf.** `services.email`, `services.payment`, `services.sms` exist as thin stubs. No real shelf yet — no auth, no push notifications, no file upload, no maps.
- **Phase 6: Sensors.** Not started.
- **Phase 7: Shells.** Currently one shell: a plain iframe. Phone chrome, tablet, terminal, conversational — all deferred.
- **Phase 8: Lower-frequency spine patterns.** Frame-loop (games), document+undo (editors), declarative diff/apply (IaC), pure-compute pipeline — none built yet.
- **Phase 9: Graceful degradation / stub concept.** Not implemented. The `stub` handle isn't in the runtime yet.
- **Phase 10: The agent.** Not started. The intake conversation, manifest, code generation, and iteration loop are all future work.
- **Phase 11: Library system.** Not started.

## Best next steps

The order matters. The contract and harness need to be right before anything else is built against them.

### 1. Freeze the Phase 1 runtime contract

The Phase 0 contract held up well. The main thing needed before it can be frozen:

- Define the `host` interface. This is the trickiest part — it varies per shell, and no example exercises it yet. Even a minimal `host` for the viewport shell (nothing) and the headless shell (stdin/stdout) would nail the shape.
- Decide the `Math.random()` containment strategy.
- Give seeding a real home (manifest-driven, not an app call).
- Document the postMessage bridge shape (already partially built via `radix:dump` / `radix:reset`).

### 2. Build the simulation console in the app shell

This is the highest-leverage near-term addition. Right now the only cockpit tool is the DB inspector. Adding clock controls (play/pause/step/fast-forward), an event log, and an event injector to the shell would make the existing examples dramatically more useful to actually use and demo. It also validates the postMessage bridge design before the real runtime is built on top of it.

### 3. Build the generic schema-driven store (Phase 2)

Right now every example hand-writes its seed. A real schema definition (strict core + open extension layer, as designed in `notes.md` §6) would let examples declare their data model once, have the store configure itself, and get auto-CRUD for free. This is the single biggest leverage point in the system.

### 4. Build more examples targeting gaps

The current examples are strong on CRUD and basic world-simulation. Gaps worth targeting:

- A relational example that exercises `include` properly and surfaces whether the current query model is enough.
- A headless API example (app 27) that forces the `host` question for the headless shell.
- A sensor-based example (fitness tracker) to start thinking about the sensor primitive.
- One of the graceful-degradation examples (collaborative editor) to force the `stub` design.

### 5. Defer the agent until the library is solid

The plan is right about this: the agent should come last, because it orchestrates a library that has to exist first. The examples are a good proxy — until they're comfortable to build and feel like "mostly configuration," the library isn't ready for an agent to use.

## Things to keep in mind going forward

- **The DB inspector panel is a good model for the console.** It uses postMessage cleanly and the pattern (shell sends a command, iframe responds with data) should extend to clock controls and event injection.
- **The `wrapPrototype` / `wrapReactApp` split is intentional.** Smoke tests don't need the runtime; Phase 0+ examples do. Keep this distinction as the runtime grows.
- **Visual variety matters.** The plan flags this explicitly. Generated UIs should feel genuinely different from each other. Worth an explicit mechanism once code generation starts.
- **The embedded-in-host shell (VS Code extension, macOS tray, browser extension) is the highest-risk category.** Deliberately defer past v1.
